# 06 使用场景分析

> 状态：草稿（待对齐） ｜ 最后更新：初始化

> 前提（见 01 号 Decision Log）：所有场景都发生在「**用户给 AI 派活 → AI 完成后向用户汇报**」这一环。AI 是下属，用户是领导。下面的「场景」即 AI 完成不同类型任务后的汇报形态。

## 1. 场景分类总览

| 类别 | 典型情形（AI 干完活后） | 推荐结构 |
| --- | --- | --- |
| 任务完成汇报 | AI 做完一个任务（如写完代码/查完资料） | BLUF + 闭环四要素 |
| 决策请示 | AI 遇到岔路口，需要用户拍板 | 选项 + 推荐 |
| 问题/风险上报 | 任务中出现报错、阻塞、做不下去 | 危机汇报结构 |
| 阶段性进展 | 长任务做到一半，阶段汇报 | 闭环四要素 |
| 成果复盘 | 一个较大任务收尾总结 | STAR |
| 方案提报 | AI 给出实施方案待确认 | 金字塔 + 5W2H |
| 信息答复 | 用户直接提问，AI 直答 | BLUF |

## 2. 典型场景详述

### 场景 1：任务完成汇报（最高频）
- **情形**：AI 完成用户布置的任务（如「把这个函数重构了」「查完了竞品资料」）。
- **结构**：结论（任务是否完成 / 整体状态）→ 关键产出/改动 → 风险或注意事项 → 下一步/待办。
- **要点**：先说「做完了没、结果如何」，而不是从头复述执行过程。过程细节放在后面分层。

### 场景 2：决策请示
- **情形**：AI 执行中遇到多种可行路径，需要用户定夺（如「有两种实现方案」「需求有歧义」）。
- **结构**：结论性建议 → 2–3 个选项（各列利弊）→ 明确推荐项 + 理由 → 需要用户决策的点。
- **反模式**：把问题原样抛回（「你想怎么做？」）而不给建议。

### 场景 3：问题/风险上报（危机汇报）
- **情形**：任务中出现报错、被阻塞、无法完成或结果不及预期。
- **结构**：影响（出了什么问题、影响多大）→ 已采取的措施/尝试 → 原因 → 后续方案 → 需要的支持。
- **要点**：先讲影响和「已经试过什么」，再讲原因，避免一上来一堆报错堆栈或甩锅式解释。

### 场景 4：成果复盘
- **触发**：「总结一下这个项目」。
- **结构**：STAR（情境-任务-行动-结果）+ 关键经验 + 后续建议。
- **要点**：结果尽量可量化。

### 场景 5：方案提报
- **触发**：「给个方案」「写个提案」。
- **结构**：核心建议 → 方案要点（金字塔）→ 5W2H 完整性 → 风险与预案 → 所需资源。

### 场景 6：信息答复
- **触发**：领导直接提问。
- **结构**：BLUF 直接回答 → 必要的支撑 →（如有）补充提示。

## 3. 触发与不触发（边界）

本项目采用**项目内常开**：默认 AI 每次向用户汇报都用汇报体。

仍需保留的例外（避免形式主义）：
- 用户临时要求「用普通方式说」「别这么正式」→ 降级/关闭。
- 闲聊、情感陪伴等非任务场景 → 不硬套（即便常开，也不应把寒暄写成汇报）。

> 详见 10 号文档《反模式与使用边界》。

## 4. 场景—结构映射速查表

```
进展类   → 闭环四要素
决策类   → 选项 + 推荐
危机类   → 影响 → 措施 → 原因 → 方案 → 支持
复盘类   → STAR
方案类   → 金字塔 + 5W2H
问答类   → BLUF
```

## 5. 端到端示例（AI 完成编码任务后的汇报）

> 用户（领导位）给 AI 派活：「帮我给登录接口加上限流。」AI 做完后这样汇报：

**汇报体输出（示意）：**

> 结论：已完成登录接口限流，按每 IP 每分钟 5 次的策略上线，验证通过。
>
> 关键改动：
> 1. 在 `auth` 中间件接入限流器，超限返回 429。
> 2. 配置项可调（默认 5 次/分钟），已写入配置文件。
>
> 需注意的风险：限流阈值是我按常见默认值设的，生产高峰可能偏严，建议您确认数值。
>
> 下一步：如阈值 OK，我可以补上对应的单元测试；如需按用户维度（而非 IP）限流，我再调整方案。

**对比——默认风格通常会**：从头复述「我先打开了文件、然后我加了一个函数、接着我又……」，把结论和需要用户拍板的点埋在最后。
